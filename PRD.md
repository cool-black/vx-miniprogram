# 雅思口语微信小程序 v0 验证版 PRD

## 1. 这版到底要验证什么

这一版不是为了做一个完整产品。

这一版只验证一个最核心的事实：

`一个完全不会开口的雅思口语初学者，在得到非常低门槛的引导后，是否愿意说出第一段回答，并愿意根据反馈立刻再说一次。`

如果这个事实不成立，后面的题库、打卡、分享、历史记录、个人中心都没有意义。

所以这版不追求完整，只追求把最小闭环做通。

## 2. 目标用户

### 2.1 核心用户

- 雅思口语初学者
- 刚开始准备，几乎没有稳定开口练习
- 最大问题不是没资料，而是不知道怎么开始说
- 对长课程和复杂学习计划有压力

### 2.2 用户真实状态

这类用户看到题目时常见反应是：

- 我不知道第一句怎么说
- 我怕自己说得太差
- 我想练，但总是一拖再拖

所以这个版本要解决的，不是“提升很多分”，而是“让他真的开口一次”。

## 3. 核心假设

### 3.1 产品假设

如果我们把口语练习拆成一个足够简单的闭环：

1. 给一道非常容易开始的题
2. 给一句可直接套用的回答骨架
3. 让用户说 30 到 45 秒
4. 给非常短、非常具体的反馈
5. 立刻鼓励用户再说一次

那么一部分原本不开口的用户，会完成第一次录音，并愿意立刻重答一次。

### 3.2 这版不验证的事

这版暂时不验证：

- 用户会不会连续 7 天回来
- 用户会不会分享传播
- 用户会不会为它付费
- 用户需不需要完整题库
- 这个产品能不能覆盖 Part 2 或 Part 3

这些都更靠后。

## 4. 产品范围

### 4.1 必须做

这版只保留下面这个最小闭环：

`进入 -> 看题 -> 录音 -> 收到反馈 -> 再答一次`

### 4.2 明确不做

这一版不做：

- 个人中心
- 历史记录
- 连续打卡
- 分享卡片
- 邀请好友
- 收藏
- 排行榜
- 社区
- 完整题库
- 多科支持
- Part 2 和 Part 3

### 4.3 题目范围

只准备 `10 到 20 道` 雅思口语 Part 1 高频题。

原因很简单：

- 足够测试闭环
- 足够覆盖多个常见主题
- 不会把时间浪费在题库建设上

示例题类：

- Hometown
- Work or study
- Weekends
- Music
- Friends

## 5. 产品形态

### 5.1 一句话定义

一个帮助雅思口语初学者“第一次张口”和“立刻重说一次”的微信小程序。

### 5.2 核心价值

不是教完整雅思口语。

而是把用户从：

`我不知道怎么说`

推到：

`我先说出来，再改一次`

这就是这版的全部价值。

## 6. 用户流程

### 6.1 主流程

1. 用户进入首页
2. 看到当天给出的一道题
3. 看到一条超短回答骨架
4. 点击开始录音
5. 完成 30 到 45 秒作答
6. 系统返回三条超短反馈
7. 系统给一个简短参考答案
8. 用户点击“再答一次”
9. 用户完成第二次作答

### 6.2 体验目标

第一次使用的用户，应该能在 `3 分钟内` 完成第一轮作答。

看完反馈后，用户应该能在 `30 秒内` 理解自己下一次该改什么。

## 7. 页面设计

这版只做 3 个页面。

### 7.1 首页

页面目标：
让用户不做选择，直接进入练习。

页面元素：

- 一道题
- 一条回答骨架
- 一个开始录音按钮

首页不应该出现多余入口。不要把用户带去别的地方。

### 7.2 录音页

页面目标：
帮助用户完成第一次开口。

页面元素：

- 题目常驻展示
- 回答骨架提示
- 录音按钮
- 30 到 45 秒倒计时

规则：

- 页面尽量干净
- 权限请求要清楚
- 不制造考试压力感

### 7.3 反馈页

页面目标：
让用户知道“下一次该怎么说得更好一点”。

页面元素：

- 总评一句话
- 三条短反馈
- 一个参考答案
- 一个“再答一次”按钮

反馈页不是报告页。不要做复杂分析。

## 8. 反馈策略

### 8.1 只保留 3 个维度

- 是否切题
- 是否够长
- 是否自然

### 8.2 反馈形式

每个维度只给一句非常短的话。

示例：

- 切题：你有回答问题本身，没有明显跑题
- 长度：答案偏短，可以多补一个原因
- 自然：能听懂，但有些表达有点像中文直译

### 8.3 反馈原则

- 说人话
- 可立刻执行
- 不吓人
- 不做复杂打分

### 8.4 不做的反馈

- 发音逐词纠错
- 细粒度语法批改
- 官方口语四项完整评分
- 冗长评语

## 9. 成功指标

### 9.1 这版最重要的 3 个指标

- 首页到首次录音提交的完成率
- 反馈页到“再答一次”的点击率
- 第二次录音完成率

### 9.2 建议观察阈值

如果是早期小规模测试，这版至少要看到：

- 首次录音提交率大于 50%
- “再答一次”点击率大于 30%
- 第二次录音完成率大于 20%

这些数字不是行业标准，只是帮助判断这个闭环有没有生命力。

## 10. 内容设计

### 10.1 单题内容结构

每道题只需要这些内容：

- 题目原文
- 主题标签
- 回答骨架
- 2 到 3 个关键词
- 一个参考答案

### 10.2 题目 canonical schema

所有种子题目都必须遵循同一个固定结构，不能让前端、后端、内容录入各自定义字段。

建议结构如下：

```json
{
  "id": "part1_hometown_001",
  "part": "part1",
  "topic": "hometown",
  "prompt": "Do you like your hometown?",
  "hint": "Yes, I do. The main reason is... Also...",
  "keywords": ["comfortable", "familiar", "food"],
  "sample_answer": "Yes, I do. I really like my hometown because it is comfortable and familiar to me."
}
```

这个结构要作为题目内容的唯一来源，供前端展示、后端生成反馈、种子内容录入共同复用。

### 10.2 内容要求

- 参考答案足够短
- 接近真实口语
- 不追求复杂词汇
- 优先帮助用户“模仿着先说出来”

## 11. 技术最小方案

### 11.1 必要链路

1. 小程序展示题目
2. 用户录音上传
3. 音频转文字
4. 基于题目和转写结果生成结构化反馈
5. 反馈页展示结果
6. 支持同题再录一次

### 11.2 后端拥有反馈契约

AI 反馈不能直接暴露给前端，必须由后端统一整理成固定 response schema，前端只负责渲染。

推荐接口形态：

```text
Mini Program
  -> POST /practice-attempts
     -> store audio
     -> transcribe
     -> generate feedback JSON
  <- {
       question,
       transcript,
       feedback: {
         overall,
         relevance,
         length,
         naturalness
       },
       sample_answer,
       retry_token
     }
```

实现原则：

- 后端是字段定义的唯一来源
- 前端不直接消费模型原始输出
- AI 输出先被规范化，再返回前端

### 11.2 这版技术重点

- 录音稳定
- 转写速度可接受
- 反馈结构稳定
- 再录一次流程顺

### 11.3 schema 校验与失败兜底

后端必须对转写结果和反馈结果做结构校验，不能把不完整或脏数据直接交给前端。

至少覆盖以下失败情况：

- 录音权限被拒绝
- 音频上传成功，但转写失败或超时
- 转写成功，但 AI 返回结果不符合 schema
- 用户点击“再答一次”后，没有正确沿用同一 question id

最小要求：

- 转写失败时，给用户明确错误提示和重试入口
- 反馈 schema 不合法时，后端返回安全默认结构或明确失败结果
- 前端不因为字段缺失而白屏

### 11.4 提交到反馈的时间预算

从用户提交录音到显示反馈，需要有明确的体验预算。

建议标准：

- 目标：5 秒内看到反馈
- 超过 8 秒：前端显示“还在分析你的回答”
- 超过 15 秒：给出明确失败提示和重试入口

这个预算不是性能优化细节，而是产品体验边界，开发时需要按这个标准判断方案是否可接受。

### 11.5 这版技术上不追求的东西

- 音频精细分析
- 复杂学习数据报表
- 多端同步
- 大规模内容管理后台

## 12. MVP 功能清单

### 12.1 必须有

- 微信登录或最简身份识别
- 固定题目下发
- 回答骨架展示
- 录音
- 上传
- 转写
- 反馈生成
- 参考答案展示
- 再答一次

### 12.2 不做

- 用户成长体系
- 分享
- 打卡
- 历史记录
- 收藏
- 搜题
- 自选题库

## 13. 风险与应对

### 13.1 风险一，用户还是不敢录音

如果用户连第一下都不点，产品就没有开始。

应对：

- 首批题目选极简单题
- 回答骨架尽量直接可套
- 文案降低心理压力，比如“先说一句也算开始”

### 13.2 风险二，反馈太空

如果反馈像套话，用户不会信。

应对：

- 强约束反馈格式
- 控制在 3 个维度
- 用短句输出，不写大段分析

### 13.3 风险三，第二次作答动力不足

如果用户看完反馈却不想重答，闭环就断了。

应对：

- 反馈直接指向下一次修改点
- “再答一次”按钮足够突出
- 参考答案长度适中，便于立刻模仿

## 14. 开发计划

这版不需要两周大计划，按最短路径做。

### Day 1

- 定题目格式
- 准备首批 10 到 20 道题
- 画出 3 个页面的最简结构

### Day 2

- 接通录音能力
- 完成题目展示和录音页

### Day 3

- 接通上传和转写
- 跑通从录音到文本的链路

### Day 4

- 接入反馈生成
- 固定反馈返回结构

### Day 5

- 完成反馈页
- 跑通“再答一次”完整流程

### Day 6

- 小范围找种子用户测试
- 重点观察是否真的有人愿意重答一次

## 15. 验收标准

只有满足下面这些，才算 v0 可测：

- 用户可以从首页顺利进入录音
- 用户可以完成一次录音提交
- 系统能返回结构化反馈
- 用户可以直接进入第二次作答
- 整个流程在移动端上不别扭

## 16. 下一阶段触发条件

只有在 v0 验证出用户真的愿意开口并愿意重答一次后，才进入下一阶段。

下一阶段才考虑加的内容：

- 连续打卡
- 历史记录
- 分享卡片
- 题量扩充到 100+
- 个性化练习

## 17. 结论

这版产品不是一个“雅思学习平台”。

它只是一个非常小、非常具体的验证器。

它要验证的不是市场想象，而是一个更硬的问题：

`我们能不能把一个不敢开口的人，推到真的开口，并愿意立刻再来一次。`

先把这个做成。别急着装修。
