# 雅思口语微信小程序项目阶段报告

- 报告日期：2026-04-10
- 项目阶段：MVP 闭环已打通，进入“验证前强化”阶段
- 仓库地址：https://github.com/cool-black/vx-miniprogram

## 1. 项目目标回顾

根据 [PRD.md](d:\chi\ai\test-gstack\PRD.md)，v0 的核心目标不是做完整学习平台，而是验证一个更小但更关键的问题：

一个几乎不会主动开口的雅思口语初学者，在足够低门槛的引导下，是否愿意完成第一次口语回答，并在看到反馈后愿意立刻再答一次。

因此，这一阶段的重点不是题库规模、运营功能或长期留存，而是最小闭环是否真实成立。

## 2. 当前阶段结论

从当前仓库实现看，项目已经完成核心 MVP 闭环：

`进入 -> 看题 -> 录音 -> 收到反馈 -> 再答一次`

前后端都已具备基本联调能力，已经不是纯脚手架或页面原型，而是一个可以本地演示、可继续强化后进入种子测试的小版本。

当前最重要的结论是：

- “能不能做出来”这一阶段已经完成
- 接下来要解决的是“稳不稳定、信不可信、能不能拿到有效验证数据”

## 3. 已完成成果

### 3.1 前端最小闭环已完成

当前小程序已经完成 3 个核心页面：

- 首页：展示今日题目、回答骨架、关键词，并进入录音页
- 录音页：处理录音权限、倒计时、录音结束、识别文本、提交反馈
- 反馈页：展示总评、三维反馈、参考答案，并支持“再答一次”

这意味着 v0 的核心用户路径已经可走通。

### 3.2 后端接口已完成基础实现

后端已具备以下接口能力：

- `GET /health`
- `GET /questions/today`
- `GET /asr/tencent/session`
- `POST /practice-attempts`

其中 `POST /practice-attempts` 已完成：

- 题目校验
- 音频持久化
- 转写处理
- 反馈生成
- 返回统一结构化结果

### 3.3 语音识别已具备双模式兜底

当前项目支持两种 transcript 路径：

- `manual`
- `tencent`

这意味着即使腾讯语音识别失败，用户仍可手动补全文本继续提交，避免主流程完全中断。

### 3.4 反馈服务具备真实服务与 fallback 机制

后端反馈生成支持：

- MiniMax
- OpenAI
- mock fallback

当前实现不是把模型原始结果直接交给前端，而是先做结构整理和安全兜底，这与 PRD 中“后端拥有反馈契约”的方向一致。

### 3.5 基本工程协作条件已具备

当前项目已经：

- 完成 Git 初始化
- 推送到 GitHub
- 配置基础 `.gitignore`
- 排除本地私有配置与运行时数据

这意味着项目已经具备继续协作和版本推进的基本条件。

## 4. 当前未完成项

### 4.1 自动化测试仍为空白

仓库当前没有看到系统化的：

- 后端接口测试
- schema 校验测试
- 小程序页面状态测试
- 冒烟自动化测试

这意味着当前“可运行”主要依赖人工验证，稳定性结论还不够硬。

### 4.2 核心验证指标尚未真正落地

PRD 中最关键的验证指标包括：

- 首页到首次录音提交的完成率
- 反馈页到“再答一次”的点击率
- 第二次录音完成率

但当前代码中还没有完整的行为埋点或统一的结果汇总结构，因此即使开始找用户试，也很难得到清晰的验证结论。

### 4.3 数据存储仍是临时方案

当前作答结果主要落在本地 `.runtime` 和 `jsonl` 文件中，适合本地开发验证，但还不适合作为更正式的小范围测试数据底座。

### 4.4 错误处理还没有完全产品化

目前前端已经覆盖部分关键异常路径，比如：

- 录音权限拒绝
- 网络请求失败
- 识别失败后手动补文本
- 提交超时提示

但前后端错误码、提示文案和失败恢复动作还没有完全统一，仍需要一轮系统整理。

### 4.5 视觉与交互仍处于“基础可用”阶段

当前 UI 已满足功能演示，但还不算验证阶段的最优状态。对于“是否愿意点开第一次录音”这种心理门槛很高的行为，页面语气、视觉压力和交互引导仍然会明显影响结果。

## 5. 风险判断

### 5.1 最大风险不是技术跑不通，而是验证噪音过大

如果在以下几项尚未稳定前就开始找用户测试：

- 真机链路不稳
- 识别时好时坏
- 超时反馈不可预期
- 埋点数据不完整

那么后续即使拿到了一些用户反馈，也很难判断到底是产品假设有问题，还是实现质量影响了结果。

### 5.2 第二次作答的驱动力仍需验证

当前流程已经支持“再答一次”，但产品真正要证明的是用户会不会自发点击它、并完成第二次提交。这个点要靠真实数据验证，不能只靠主观感觉。

## 6. 阶段判断

建议将当前项目状态定义为：

`阶段 1：MVP 可运行验证版完成`

下一步进入：

`阶段 2：验证前强化与种子测试准备`

这个阶段的目标不是扩功能，而是把当前 MVP 从“能演示”推到“能用于小规模验证”。

## 7. 下一阶段目标

下一阶段的核心目标应聚焦在 4 件事：

1. 提升主流程稳定性
2. 提升反馈可信度和失败恢复能力
3. 建立核心验证指标的采集机制
4. 完成真机和种子测试准备

只有这些基础打稳，后续的用户测试结论才有意义。

## 8. 本阶段总结

当前项目已经完成了最重要的一步：把产品假设对应的最小闭环真正做出来了。

这一步价值很高，因为它把问题从“想不想做”推进到了“能不能验证”。现在不应该急着扩历史记录、打卡、分享或大题库，而应该集中资源把当前这个闭环做稳、做顺、做可观测。

如果下一周期能补齐稳定性、埋点和真机体验，这个项目就可以进入一次真正有价值的种子测试。
